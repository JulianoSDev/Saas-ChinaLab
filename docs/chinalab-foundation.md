# ChinaLab Foundation

## Vision

ChinaLab is not a generic bot, a dry directory, or a standalone calculator.

ChinaLab is the decision layer between the link and the purchase.

The product exists to help the user:
- discover better
- evaluate with more confidence
- plan with fewer mistakes
- follow changes without depending on memory

The goal is not only to answer commands.
The goal is to make the user feel:

`if I am going to buy, I check ChinaLab first.`

## Product Principles

### 1. Build assets, not isolated features

Every delivery should strengthen something reusable:
- item identity
- seller identity
- normalized link
- snapshots
- watch
- reputation
- analytics
- history

### 2. Trust before sophistication

The product should not look smart without a reliable base.

### 3. Score only with explainability

If the system cannot explain why something is good, bad, or risky, it is not ready to become a score.

### 4. Retention before gamification

Watchlist, alerts, history, and return matter more than badges, rankings, and cosmetic social features.

### 5. Community as intelligence, not just engagement

Findings, reviews, QC, and community signals should improve real decision quality for other users.

### 6. Monetize decision, not “AI”

ChinaLab should monetize:
- saved time
- reduced error
- better decisions
- better conversion

### 7. Every delivery must serve beginner and advanced users

- beginners must not get blocked
- advanced users must stay fast
- responses should guide the next useful action

## System Capabilities

### Discover

Help the user find items, sellers, opportunities, and entry points.

### Evaluate

Help the user understand cost, context, confidence, and practical value.

### Plan

Help the user organize a purchase, a haul, and practical next steps.

### Follow

Help the user track price, seller, link, and opportunity changes over time.

These four capabilities matter more than any single command.

## Official Roadmap

### Phase 0 — Product Spine

#### Goal

Create the minimum structural base that allows the product to grow without twisting.

#### Invisible deliverable

- identity conventions
- internal seller/item/link consistency
- minimum persistence structure for future growth
- base analytics

#### Visible deliverable

- no major hero feature
- improved consistency across responses and internal objects

#### Main KPI

Consistency of internal references between link, seller, and item.

#### Proof event

The team can launch a persistent feature without unnecessary structural rework.

#### Kill list

- score
- ranking
- gamification
- public web
- cosmetic AI

### Phase 1 — Trust Data Layer

#### Goal

Make ChinaLab feel concretely useful and trustworthy.

#### Invisible deliverable

- useful persistence
- watch foundations
- first snapshots
- more reliable links between entities

#### Visible deliverable

- stabilized `/vendedor`
- watchlist V1
- lightweight review MVP
- clearer seller/item reading

#### Main KPI

Save/watch rate after seller or item consultation.

#### Proof event

A user checks a seller or item and then saves, follows, or returns later.

#### Kill list

- opaque score
- public ranking
- large dashboard
- overblown AI

### Phase 2 — Retention Engine

#### Goal

Create habit and return.

#### Invisible deliverable

- notification engine
- more reliable snapshots
- passive dead-link checking
- saved searches

#### Visible deliverable

- useful alerts
- basic history
- simple personal feed

#### Main KPI

Return rate triggered by alert or watch behavior.

#### Proof event

A user returns because the system signaled something important.

#### Kill list

- heavy gamification
- big web push
- aggressive monetization

### Phase 3 — Intelligence Layer

#### Goal

Turn data into practical decision support.

#### Invisible deliverable

- explainable rules
- comparison logic
- trust signals
- recommendation logic

#### Visible deliverable

- explainable score alpha
- comparisons
- `/trending`
- haul planner
- useful snapshots

#### Main KPI

Action rate after comparison or recommendation.

#### Proof event

A user takes a practical decision after seeing the intelligence layer output.

#### Kill list

- magic score
- unexplained rating
- AI for its own sake

### Phase 4 — Community Intelligence

#### Goal

Turn community into a competitive advantage.

#### Invisible deliverable

- reputation
- validation
- reuse of community-generated content

#### Visible deliverable

- `/qc`
- public collections
- haul feed
- more useful reviews

#### Main KPI

Share of community content that improves decision quality for other users.

#### Proof event

One user’s content changes another user’s decision.

#### Kill list

- empty engagement
- social features without utility
- badges without signal

### Phase 5 — Monetization Engine

#### Goal

Capture value without damaging trust.

#### Invisible deliverable

- attribution paths
- conversion analytics
- plan rules

#### Visible deliverable

- creator tools
- advanced watchlist/alerts
- individual pro plan
- creator plan
- first team utilities

#### Main KPI

Revenue conversion without trust or usage collapse.

#### Proof event

A user pays because ChinaLab reduces error or improves efficiency.

#### Kill list

- premature paywalls
- selling empty promises
- monetization before habit exists

### Phase 6 — Web Platform

#### Goal

Expand distribution without losing the product’s core identity.

#### Invisible deliverable

- coherent API
- reusable structure
- auth
- indexing

#### Visible deliverable

- seller pages
- item pages
- reviews
- watchlist
- history
- search

#### Main KPI

Useful web traffic plus recurring usage conversion.

#### Proof event

A user discovers ChinaLab on the web and continues using the ecosystem.

#### Kill list

- dead catalog
- web before trust/data
- pretty front-end without depth

## Decision Criteria For New Ideas

Every new idea should answer “yes” to most of these questions:
- does this create a reusable asset?
- does this improve real decision quality?
- does this reduce error or friction?
- does this increase user return?
- does this strengthen trust?
- does this strengthen moat?
- does this fit the current phase?
- does this prevent future rework?

If it does not pass this filter, it should not enter the roadmap.
